# CI/CD as the Control Plane for AI-Assisted Engineering

AI coding assistants and autonomous agents accelerate code generation dramatically. CI/CD becomes the mandatory enforcement layer that keeps that speed from outpacing quality — it is the point where all code, regardless of origin, is validated against the same objective standards.

## Core principle

**AI-generated code must pass the same gates as human-written code.** Bypassing CI because "it looks fine" or "the AI generated it" removes the only reliable check that the change integrates correctly with the rest of the system.

## Why CI/CD is more important with AI assistance

- AI models can generate plausible but incorrect logic, hallucinate APIs, or introduce subtle security flaws
- AI agents working autonomously have no situational awareness of system-wide invariants — only automated tests can encode those
- Code volume increases; human reviewers have less time per line; automated checks absorb the gap

## Key practices

### Treat AI output like any other input
- Same lint, type check, test, and security scan gates apply
- Do not create lighter-weight pipelines for AI-assisted PRs
- Code review remains mandatory — AI generates a first draft, not a final answer

### Shift-left feedback loops
- Pre-commit hooks (lint, format, secrets scan) give immediate signal before code ever reaches the remote
- Local validation should mirror CI — AI tools should be configured to run local checks before submitting
- Fast CI (<10 min) is especially important: AI can iterate on failures, but only if feedback is quick

### Security gates matter more
- **Secrets detection** — AI models may reproduce credential patterns from training data
- **SAST** — AI-generated code can introduce injection vulnerabilities or insecure defaults
- **Dependency scanning (SCA)** — AI may suggest outdated or vulnerable package versions
- **License compliance** — AI can pull in dependencies without considering license compatibility

### Integration and contract tests become the source of truth
- Unit tests on AI-generated code can pass while the broader integration breaks
- AI lacks system context; integration tests encode it
- Contract tests guard API boundaries that AI changes might silently violate

### Version and review prompts in agentic workflows
- When an AI agent generates a PR autonomously, include the prompt or task description in the PR body
- This makes the intent auditable and reproducible
- Treat significant prompt changes as code changes — they alter behaviour

## Pipeline design principles

```
pre-commit → build → unit test → integration test → security scan → deploy
```

- **Fail fast**: place cheap, high-signal checks (lint, type check) first
- **Determinism**: AI output can vary across runs; tests must be stable and not flaky
- **Separation of concerns**: do not bundle deployment into the validation phase
- **Mandatory human approval** for merges generated by autonomous agents — CI validates, humans decide

## Observability of AI contributions

Track where AI assistance is used so quality signals can be correlated:

- Label AI-assisted PRs (convention or tooling)
- Monitor defect rates and revert rates for AI-assisted vs human code
- Use metrics to identify where AI assistance is high-value vs where it introduces noise

This data informs where to invest in tighter automated checks and where AI tooling needs calibration.

## Agentic workflow integration

When AI agents run end-to-end (open issue → write code → open PR):

1. Agent submits PR with full context in the description
2. CI runs the full standard pipeline — no exceptions
3. Human reviewer approves (or rejects with feedback for the agent to act on)
4. Merge only after CI passes and human approval is given

CI is the control plane here: it provides objective, reproducible validation that no human reviewer can consistently replicate at the volume AI agents produce.

## Target model

Use a **trunk-based, small-batch delivery model** with strong automated gates. DORA’s AI work highlights that AI success depends on foundations like strong version control, quality internal platforms, small batches, and AI-accessible internal context. ([Dora][1])

### 1. Repo and branching

For a large web app:

```text
main
 ├─ short-lived feature branches
 ├─ AI agent branches
 └─ release tags / deployment markers
```

Avoid long-lived team branches. Every team and every AI agent works through pull requests.

Rules:

* no direct commits to `main`
* small PRs, ideally under 300–500 changed lines
* every PR linked to an issue/story
* every PR includes acceptance criteria
* AI-generated PRs must be labelled, for example `ai-authored`, `ai-assisted`, or `human-authored`
* CODEOWNERS required for impacted areas

GitHub’s coding agent model also works through branches and pull requests rather than bypassing normal review. ([GitHub Docs][2])

### 2. Pipeline shape

Use a **layered pipeline**, not one giant slow pipeline.

```text
PR opened
  ↓
Fast checks: lint, typecheck, unit tests, dependency checks
  ↓
Affected-area tests
  ↓
Storybook/component tests
  ↓
Contract tests
  ↓
Security and supply-chain checks
  ↓
Preview environment
  ↓
Human + AI review
  ↓
Merge to main
  ↓
Build once, promote artifact
  ↓
Progressive deployment
  ↓
Observability + rollback automation
```

### 3. PR checks

For a modern web app, every PR should run:

```text
Code quality
- format
- lint
- TypeScript typecheck
- architectural boundary checks
- dead code / circular dependency checks

Testing
- unit tests
- affected component tests
- affected BFF/API tests
- contract tests
- accessibility checks
- visual regression for changed UI
- smoke E2E on preview env

Security
- SAST
- dependency vulnerability scan
- secret scan
- license policy
- container scan
- SBOM generation
- provenance/signing
```

For supply chain, align with **SLSA-style provenance**, signed artifacts, pinned dependencies, and reproducible builds where practical. SLSA is the commonly referenced framework for securing software supply chains through provenance and build integrity. ([Google][3])

### 4. AI-specific controls

AI coding agents change the risk profile. They increase throughput, but can also increase review load, inconsistency, and instability if the pipeline is weak. DORA’s 2025 AI research focuses on AI as something amplified by existing engineering capability rather than a replacement for it. ([Google Cloud][4])

Add these controls:

```text
AI PR requirements
- issue must contain clear acceptance criteria
- PR must include implementation summary
- PR must include test evidence
- PR must explain risk and rollback
- PR must declare generated/modified files
- no AI agent can self-approve
- human owner required for merge
```

Use an **AI reviewer**, but never as the only reviewer. The AI reviewer should check:

* acceptance criteria coverage
* missing tests
* security smells
* architectural violations
* inconsistent patterns
* large unexplained changes
* frontend accessibility
* performance regressions

### 5. Risk-based gates

Do not make every change go through the same heavyweight process. Use **risk-based CI/CD**.

Example:

```text
Low risk
- copy change
- isolated component
- no API change
- no auth/payment/customer data
→ fast checks + preview + one review

Medium risk
- new UI flow
- BFF change
- shared component change
→ full affected tests + contract tests + two reviewers

High risk
- auth
- payments
- customer data
- security controls
- shared shell/navigation
- design system core
- API contract breaking change
→ full regression + security review + architecture owner approval
```

This is especially important when AI agents are operating at speed.

### 6. Preview environments

Every PR should get an ephemeral preview environment:

```text
https://pr-1234.app.example.com
```

Include:

* deployed frontend
* mocked or sandbox BFF
* seeded test data
* Storybook link
* test report
* accessibility report
* visual diff report
* bundle size report

This lets product owners, designers, reviewers, and agents inspect the actual change quickly.

### 7. Monorepo or polyrepo

For many teams on one large web app, I would favour a **monorepo** if the organisation can support it well.

Benefits:

* easier affected builds/tests
* shared standards
* atomic changes across shell, MFEs, components, and BFF
* better agent context
* easier dependency governance

Use Nx, Turborepo, Bazel, or similar affected-graph tooling.

Suggested structure:

```text
/apps
  /web-shell
  /customer-dashboard
  /admin-portal

/packages
  /design-system
  /auth
  /api-client
  /logging
  /feature-flags
  /test-utils

/services
  /bff
  /mock-api

/platform
  /ci
  /deployment
  /policy
```

### 8. Deployment model

Use **build once, deploy many**.

```text
main merge
  ↓
create immutable artifact
  ↓
deploy to dev
  ↓
promote to test
  ↓
promote to staging
  ↓
progressive production rollout
```

Production should use:

* feature flags
* canary deployment
* blue/green where needed
* automatic rollback on SLO breach
* release markers in observability
* kill switches for risky features

### 9. Governance model

You need **federated governance**, not central bottlenecks.

```text
Platform team owns:
- pipeline templates
- golden paths
- policy-as-code
- deployment platform
- observability standards

Product teams own:
- features
- tests
- service quality
- release readiness

Architecture/design system/security own:
- policies
- reusable patterns
- exception process
- audit sampling
```

Use policy-as-code so governance is embedded in CI/CD instead of handled in meetings.

### 10. Metrics

Track classic DORA metrics, but add AI-era metrics:

```text
Delivery
- lead time
- deployment frequency
- change failure rate
- MTTR

Quality
- escaped defects
- flaky test rate
- rollback rate
- rework rate
- production incident rate

AI-specific
- AI-authored PR failure rate
- AI PR rework rate
- average review comments per AI PR
- percentage of AI PRs requiring major rewrite
- policy violations by agent/tool
- test coverage delta from AI PRs
```

## Recommended operating model

The best model is:

```text
Small PRs
+ trunk-based development
+ ephemeral preview environments
+ affected tests
+ contract testing
+ policy-as-code
+ human-owned merge
+ AI-assisted review
+ progressive delivery
+ strong observability
```

The principle is simple:

**Let AI agents move fast inside a narrow, observable, policy-controlled delivery system.**

Do not slow everyone down with manual governance. Do not let agents bypass engineering discipline. Let the pipeline become the guardrail.
